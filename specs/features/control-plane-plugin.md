# 防御控制面插件 (Edge Control Plane Plugin)

## 1. 产品背景与定位
当前许多小项目、小玩家在面临恶意攻击时，由于缺乏专业的运维能力，往往直接暴露源站真实IP，导致被 DDoS 击穿或被爬虫恶意消耗资源。

**防御控制面插件** 是 KForge 平台为中小玩家打造的最具商业价值的“护城河级”核心插件。它的核心思想是：**绝对的动静分离与职责单一**。

### 核心设计原则
1. **隐藏 IP 做在网络边界**：通过 CDN/WAF/反向代理层对外暴露，源站服务器仅配置防火墙白名单，且不对外解析真实域名。
2. **插件不直接操作基础设施**：插件绝不直接修改 Nginx 或 iptables，而是作为“控制面”提供配置模型与状态开关。
3. **只做两件事**：
   - **动态下发配置**：允许哪些边缘节点回源、回源策略、端口映射。
   - **观测与熔断**：检测异常流量，触发系统自保策略（如降级为只读模式、关闭注册等）。

## 2. 核心架构设计：两层源站保护模型

### A. 边缘层 (Edge Layer - 对外唯一入口)
- CDN / WAF / 自建边缘 Nginx 节点对外提供业务域名（如 `goodspage.cn`）。
- 终端用户的 TLS 证书在边缘节点终止。

### B. 源站层 (Origin Server - 真实业务服务器)
- 防火墙白名单机制：仅允许来自边缘节点的 IP 访问。
- 所有业务端口（如 80, 443, 11000, 7880）对公网全部关闭。
- LiveKit / WebSocket 等长链接协议同样必须经过边缘层代理，严禁直接暴露源站端口。

### C. 控制面插件 (Control Plane Plugin)
作为“防御大脑”，它需要完成：
- 接收并存储边缘策略配置。
- 提供 API 供外部控制器（如 Ansible 脚本、K8s Operator、自研配置中心）拉取最新状态。
- 根据安全事件动态修改全局熔断状态。

## 3. 功能需求说明

### 3.1 配置模型管理
插件需要维护当前的边缘防御策略：
- `edgeMode`: 当前模式 (CDN 模式 / 自建边缘节点模式)。
- `whitelistIps`: 允许回源的合法边缘节点 IP 列表。
- `portMappings`: 边缘节点端口到源站端口的映射关系。
- `wafEnabled`: 是否在边缘开启 WAF 过滤。

### 3.2 控制面 API
提供 RESTful API 供外部系统交互：
- `GET /api/defense/config`: 获取当前防御配置与熔断状态（供外部脚本/网关读取以生成 Nginx 规则）。
- `PUT /api/defense/config`: 更新防御策略（仅限具有 Admin 权限的系统或管理员调用）。

### 3.3 熔断与降级机制 (Circuit Breaker)
当系统检测到异常（例如通过其他监控插件或外部 Webhook 传入的超量请求报警）时，插件会自动将系统状态切换为防御模式，下发以下开关状态：
- `readOnlyMode`: 开启全站只读（阻断写库型攻击）。
- `disableRegistration`: 关闭新用户注册（防止垃圾注册攻击）。
- `limitUploads`: 限制大文件上传。
- `rateLimitLevel`: 动态调整限流等级 (Low/Medium/High)。

## 4. 验收标准 (Acceptance Criteria)

- **AC-01 [配置读取]**: 外部脚本可通过 GET 接口获取最新的 `whitelistIps` 和 `portMappings` 数据，且格式规范，便于渲染 Nginx/防火墙规则。
- **AC-02 [安全更新]**: PUT 接口更新配置时，必须进行严格的鉴权（Admin Role / API Key）。
- **AC-03 [熔断触发]**: 支持通过接口触发全局熔断状态更新（如 `readOnlyMode = true`）。
- **AC-04 [解耦性]**: 插件代码中不得出现直接调用 `iptables`、`systemctl reload nginx` 或写入 Nginx 实体配置文件的动作。
